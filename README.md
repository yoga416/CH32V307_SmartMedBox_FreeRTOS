MAX30102 心率血氧模块开发者指南 (Developer Guide)
本指南详细拆解了 CH32V307 智能药盒项目中 MAX30102 模块的软件架构。为了保证代码的高内聚与低耦合，该模块被严格划分为四个独立层次。

开发维护人员在进行二次开发或硬件迭代时，请严格按照对应的模块进行参数微调。

模块一：硬件驱动层 (Driver & Port Layer)
文件位置：bsp_max_30102_port.c/h, bsp_max_30102_driver.c/h
核心职责：负责 I2C 底层时序、外设 GPIO 初始化、以及 MAX30102 芯片寄存器的基础配置（采样率、LED 脉宽、电流等）。

1. 当前工作机制
通信接口：使用软件模拟 I2C（SDA=PB7, SCL=PB6）。

中断引脚：INT 引脚连接至 PA1，配置为下降沿触发的外部中断（EXTI1）。

芯片配置：配置为 SpO2 模式，FIFO 满触发中断，标称采样率设定为 50Hz。

🔧 未来可能需要的改动 (How to modify)
更换引脚：若需更换 I2C 或 INT 引脚，仅需在 bsp_max_30102_port.h 中修改宏定义，无需改动任何上层逻辑。

信号太弱/太强（改变 LED 电流）：如果未来产品的外壳透光率改变，或者针对不同肤色人群测试时发现信号饱和或微弱，需在驱动初始化函数中修改 LED 发光电流寄存器 (REG_LED1_PA / REG_LED2_PA)。

改变物理采样率：若需要更高精度（如 100Hz），需在配置寄存器 REG_SPO2_CONFIG 中修改 SPO2_SR 字段。

模块二：任务调度与状态机层 (Handler Layer)
文件位置：bsp_max_30102_handler.c
核心职责：管理 FreeRTOS 任务上下文，处理中断信号量，执行“预热-采集-脱落检测”的状态机，并将纯净的数据送入算法库。

1. 核心状态机逻辑
积压清洗 (Flush)：在每次进入 max_event_calc_hr_spo2 测量前，利用 while 循环强行抽干队列中的残余指令和外部中断的残余信号量。这是解决“手指重新放上时误判两三次”的核心机制。

数据预热 (Preheating)：设定 PREHEAT_SAMPLES = 100。由于传感器刚贴合皮肤时，环境光变化及芯片自动增益调整（AGC）会导致基线剧烈漂移，系统直接丢弃前 2 秒的数据不作处理。

脱落检测 (Finger Removal)：在中断读取循环中，实时监控红外 (IR) 的直流分量。若低于阈值，立即标记 is_aborted = true，强制跳出循环，并引入 vTaskDelay(2000) 的 2 秒冷却期。

🔧 未来可能需要的改动 (How to modify)
脱落判定阈值调整：

代码位置：if (ir < 8000)

修改时机：如果产品加了厚玻璃罩，或者在强光/全黑环境下，底噪 IR 值发生变化，导致“明明没放手指却在测量”或“放了手指却提示脱落”，请大幅调高或调低 8000 这个阈值（范围通常在 5000 ~ 50000 之间）。

预热时间调整：

代码位置：#define PREHEAT_SAMPLES 100

修改时机：若发现每次出来的第一个数值依然偏差很大，可将其增加至 150（3秒预热）。

模块三：心率血氧解算核心算法层 (Algorithm Layer)
文件位置：algorithm_1.c, algorithm_1.h (基于 Maxim 官方库)
核心职责：执行交直流分离、4点滑动平均、寻找波峰波谷、R值计算及血氧查表。

1. 深度优化机制 (本项目核心资产)
时钟校准 (Clock Calibration)：MAX30102 的内部 RC 振荡器存在温漂和出厂偏差（约±10%）。代码中将 #define FS 从 50 修正为 46，彻底消除了心率计算偏高（如107bpm 降至 99bpm）的系统性误差。

重搏波屏蔽 (Dicrotic Notch Filtering)：低采样率下，心脏泵血的回弹波容易被误判为一次新心跳。代码将 maxim_find_peaks 函数中的最小峰值间距参数从 4 强行拉高至 20，实现了精准的心率锁定。

🔧 未来可能需要的改动 (How to modify)
针对不同芯片批次的 FS 校准：

代码位置：algorithm_1.h 中的 #define FS 46

修改时机：如果未来批量生产，换了一批 MAX30102 芯片，发现心率又开始整体偏高或偏低，请用高精度示波器抓取 INT 引脚频率，或直接与医用血氧仪对比，将 FS 调整到对应值（通常在 44~52 之间）。

测量极限心率的人群：

代码位置：algorithm_1.c 中 maxim_find_peaks(..., 20, 15) 的 20。

修改时机：当前的 20 代表两波峰间距至少为 0.4 秒（极限测量心率 150 bpm）。如果药盒的使用对象是运动员（运动后心率可能达 180 bpm），必须将 20 改小（例如改到 15），否则算法会漏算心跳导致心率减半。

模块四：应用与数据过滤层 (App & Callback Layer)
文件位置：app_task.c, on_hr_spo2_calculated (回调函数)
核心职责：定时触发测量事件，接收解算结果，执行应用层二次平滑滤波，并将数据打包发送给外设（如 OLED 或 WiFi 模块）。

1. 当前工作机制
滑动平均滤波器 (Moving Average)：定义了 g_hr_history 和 g_spo2_history 数组，深度为 3。算法吐出数据后，将其推入窗口求平均，使最终输出的数值呈现类似商业手表的“平缓滑动”效果，而非剧烈跳变。

异常复位：当 Handler 层触发失败回调（传入全 0 数据）时，触发 memset 彻底清空滑动历史窗口，防止旧的错误数据污染下一次的正常测量。

有效性截断：过滤掉 HR < 40 或 HR > 180 的离谱数据。

🔧 未来可能需要的改动 (How to modify)
改变数值反应的灵敏度：

代码位置：#define HR_SPO2_FILTER_SIZE 3

修改时机：如果希望数值变化更平滑（即便反应会慢个2-3秒），可以将 3 改为 5。如果希望 OLED 上的数字反应极快，可改为 1（关闭滤波）。

触发频率：

代码位置：app_task.c 中的 vTaskDelay(pdMS_TO_TICKS(16000));

修改时机：若业务需求变为“全天候连续监测”，需重新规划 App 层的延时逻辑，或者改为由人体红外传感器（或微动开关）触发测量，而不是死循环等待 16 秒。
